## Turn seven arrows into one prediction row

Chapter 5 built overlapping input–target views for training a sequence model. A
bigram table needs a different view of the same data: the **original training
documents**. If we counted through overlapping windows, a transition appearing in
two windows would receive two votes. Here every source transition must contribute
exactly once.

Use this fixed vocabulary order throughout the chapter:

| Token | `BOS` | `EOS` | `A` | `B` | `C` |
| :--- | ---: | ---: | ---: | ---: | ---: |
| ID | `0` | `1` | `2` | `3` | `4` |

`C` belongs to the vocabulary but does not occur in these two training documents.
The documents remain separate:

```text
d1: BOS(0) → A(2) → A(2) → B(3) → EOS(1)
d2: BOS(0) → A(2) → B(3) → EOS(1)
```

Before reading on, predict four things:

1. How many arrows will be counted?
2. Which successors occur after `A`, and how often?
3. Which token should be most likely after `A`?
4. Is “`C` never followed `A`” the same case as “nothing was observed after `C`”?

The first document contributes four transitions and the second contributes three:

```text
d1: BOS→A, A→A, A→B, B→EOS
d2: BOS→A, A→B, B→EOS
```

After aggregation, the only nonzero counts are
`BOS→A = 2`, `A→A = 1`, `A→B = 2`, and `B→EOS = 2`.
For current token `A`, keep the vocabulary column order
`[BOS, EOS, A, B, C]`:

```text
counts after A = [0, 0, 1, 2, 0]
row total       = 3
```

This is a **bigram** model: the current token selects one table row, and every
vocabulary token is a possible next-token column. It deliberately forgets
everything before the current token. Normalizing the observed `A` row gives the
maximum-likelihood estimate (MLE):

$$
[0/3,0/3,1/3,2/3,0/3]=[0,0,1/3,2/3,0].
$$

So `B` is the unique most likely successor. The entry for `A→C` is exactly zero,
but the row itself is well defined because three transitions leave `A`.

Now try to normalize row `C`. Its counts are `[0,0,0,0,0]` and its total is
zero. Dividing every cell by zero does not produce a probability distribution;
the **entire MLE row is undefined**. This is different from the defined zero in
the `A` row.

Add-one smoothing (`α=1`) adds one pseudocount to every candidate before
normalization. For `A`:

```text
counts                 [0, 0, 1, 2, 0]
add 1 to every cell    [1, 1, 2, 3, 1]
denominator            3 + 1×5 = 8
smoothed probabilities [1/8, 1/8, 2/8, 3/8, 1/8]
```

For `C`, the five equal pseudocounts give denominator `0 + 1×5 = 5` and the
uniform row `[1/5,1/5,1/5,1/5,1/5]`. That row is a fallback imposed by the
smoothing rule. It is not evidence that every successor is genuinely equally
likely after `C`, and it has no unique winner.

Finally, resist the tempting shortcut of concatenating the documents. The flat
sequence `[0,2,2,3,1,0,2,3,1]` contains eight adjacent pairs and invents exactly
one observation: `EOS(1)→BOS(0)`. That pair belongs to neither source document.

## Describe document-local counting and row normalization

The complete procedure is:

$$
C_{ij}=\sum_{d\in\mathcal{D}_{tr}}\sum_{t=0}^{|d|-2}\mathbf{1}[z_t^{(d)}=i\land z_{t+1}^{(d)}=j],\quad N_i=\sum_{k\in V}C_{ik},\quad \widehat P_{\mathrm{MLE}}(j\mid i)=\frac{C_{ij}}{N_i}\;(N_i>0),\quad \widehat P_{\alpha}(j\mid i)=\frac{C_{ij}+\alpha}{N_i+\alpha|V|}\;(\alpha>0)
$$

Read the first sum from the outside in. It visits each original training document
`d` separately. Positions are zero-based, just like Rust slice indices. In a
document of length `|d|`, position `t` stops at `|d|-2`, so `t+1` is the final
valid position at most. The superscript `(d)` matters: both members of a pair
come from the same document.

Current-token ID `i` chooses a row, and candidate next-token ID `j` chooses a
column. The indicator contributes one only when the adjacent IDs at positions
`t` and `t+1` equal `i` and `j`. Summing row `i` over every vocabulary column `k`
produces its outgoing observation total `N_i`.

MLE divides observed counts by `N_i`, so it exists only when `N_i>0`. Add-α
smoothing adds `α` to all `|V|` cells. The denominator must therefore gain
`α|V|`, not merely `α`; this guarantees that the smoothed row still sums to one.
Because `α>0`, even a zero-total row has a positive denominator.

In memory, the Rust implementation stores row `i`, column `j` at a row-major
offset. The table layout is an implementation choice; the probabilistic meaning
is still “next token `j`, given current token `i`.”

## Account for every symbol

| Symbol | Meaning in this chapter |
| :--- | :--- |
| $\mathcal{D}_{tr}$ | The set of original training documents, each already wrapped with `BOS` and `EOS`. Validation and test documents are excluded. |
| $d$ | One training document. |
| $|d|$ | The number of token IDs in document $d$, including its boundary tokens. |
| $t$ | A zero-based position for which the next position is still inside $d$. |
| $z_t^{(d)}$ | The token ID at zero-based position $t$ in document $d$. |
| $i$ | The current-token ID; it selects a count-table row. |
| $j$ | A candidate next-token ID; it selects a count-table column. |
| $k$ | An index used to visit every token ID in $V$ while totaling one row. |
| $V$ | The vocabulary of possible token IDs. |
| $|V|$ | The number of token IDs in $V$, that is, the vocabulary size. |
| $C_{ij}$ | The number of observed transitions from $i$ to $j$. |
| $N_i$ | The total number of observed transitions leaving $i$. |
| $\mathbf{1}[\cdot]$ | An indicator: one when its condition is true and zero otherwise. |
| $\widehat P_{\mathrm{MLE}}(j\mid i)$ | The maximum-likelihood next-token estimate, defined only when $N_i>0$. |
| $\alpha$ | The positive pseudocount added to every candidate next token. This chapter uses $\alpha=1$. |
| $\widehat P_{\alpha}(j\mid i)$ | The add-α smoothed estimate of the probability that $j$ follows $i$. |

Two phrases that sound similar now have precise locations in the table. An
**unobserved successor after `A`** is a zero cell in row `A`. A **context with no
outgoing observations** is an entire row whose total is zero. Confusing a column
with a row is the central misconception to avoid.

## Use a transparent classical baseline without mistaking it for a strong one

Before neural language models, n-gram models estimated the next word from a
fixed-size suffix of preceding words. A bigram uses the shortest nonempty such
context: one current token. It is easy to inspect and fast to query, but two
histories ending in the same token always receive the same distribution.

[Chen and Goodman (1996)](https://aclanthology.org/P96-1041.pdf) describe the
bigram maximum-likelihood estimate as a pair count divided by its context count
and explain why an unobserved n-gram receiving zero probability is a serious
language-modeling problem. Their empirical study compares several smoothing
families. Additive smoothing is a simple member of that landscape, not the
best-performing practical choice: it moves equal pseudocount mass into every
cell and cannot express which unseen continuations are more plausible.

[Bengio, Ducharme, and Vincent (2000)](https://proceedings.neurips.cc/paper/2000/hash/728f206c2a01bf572b5940d7d9a8fa4c-Abstract.html)
describe conventional n-gram models as tables of conditional probabilities over
short contexts. Their neural approach instead learns distributed word
representations so that evidence can generalize through similarity—a capability
the independent cells of our count table do not have.

This course keeps add-one smoothing for a narrower reason: every numerator and
denominator can be checked by hand. Its weakness is visible immediately. The
smoothed `A` row assigns `1/8` not only to unseen `A→C`, but also to `A→BOS`, even
though `BOS` is reserved for document beginnings. Uniform pseudocounts know
nothing about token roles or syntax.

The papers do not prescribe our token IDs, document boundaries, `α=1`, train-only
helper, or tie order. Those are explicit teaching and implementation choices.
The runnable Rust contrast below prints both MLE and add-one estimates from the
same count table so their difference is observable rather than historical prose.

2. Derive the complete count, MLE, and add-one rows for context `A`.

4. Concatenate the documents mentally. Which exact transition is fabricated, and how many pairs would the flattened sequence contain?
5. Why would fitting from Chapter 5’s overlapping `T=3`, `S=1` windows miscount document `d1`?
6. Recalculate the smoothed `A` row with `α=1/2`.
7. Verify that both add-one rows shown in this chapter sum to one.
8. State the argmax result after `A` and after `C`, including every tie.
9. Suppose validation data contains many `A→C` transitions. Should the frozen count table change before Chapter 7 scores validation?

**Misconception check:** “`C` was not observed after `A`, therefore row `C` is
undefined” confuses a column with a row. The missing `A→C` observation is one
cell in the defined `A` row. Row `C` is undefined under MLE because no training
transition has `C` as its current token.

## Freeze the first complete next-token model for scoring

The course now has a model that accepts a current token ID `i` and returns one
probability for every candidate next token `j`. It is intentionally weak—it sees
only one-token context and smoothing cannot learn similarity—but its output has
the same essential shape as a decoder’s output distribution.

Chapter 7 will look up the probability assigned to each actual next token and
aggregate those values into negative log-likelihood and perplexity. The count
table remains frozen: training examples may be scored without adding a second
set of counts, validation is evaluation-only, and the test partition stays
unopened. No gradient is needed to fit this count table; optimization of learned
parameters belongs to later chapters.
