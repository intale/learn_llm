## Predict two `[3, 2]` tensors before reading their values

Start with the same flat owner that a small Rust model will use:

```text
shape   = [2, 3]
strides = [3, 1]
storage = [10, 11, 12, 20, 21, 22]
```

The shape describes two rows and three columns. Chapter 8's offset rule gives
row-major source offsets `[0,1,2,3,4,5]`. Before reading the reveal, write the
shape, strides, logical source-offset sequence, and nested values for each request:

1. `view.reshape(&[3,2])`
2. `view.transpose(0,1)`

Neither request may move a single `f64`. Commit to both reading orders before
continuing.

### Reveal: equal shapes, different logical orders

First, reshape the view to `[3,2]`. A reshape preserves the current logical
row-major order, so the new strides are `[2,1]`. Its coordinates visit source
offsets in the same sequence:

```text
reshape values = [[10, 11], [12, 20], [21, 22]]
offsets        = [0, 1, 2, 3, 4, 5]
```

Second, transpose axes `0` and `1`. The old column axis becomes the new row
axis. Shape `[2,3]` becomes `[3,2]`, and the matching strides `[3,1]` become
`[1,3]`:

```text
transpose values = [[10, 20], [11, 21], [12, 22]]
offsets          = [0, 3, 1, 4, 2, 5]
```

The two results have equal shapes and contain equal scalar values, yet their
coordinate meanings differ. Neither changes the owner's storage. For example,
transposed coordinate `[2,1]` and source coordinate `[1,2]` both select offset
`5` and therefore the same stored `22.0`.

A view is not a copy. It is a checked interpretation containing shape, strides,
and a base offset while the original `Tensor` remains the owner. This distinction
is the chapter's central prediction: **same storage does not mean same logical
order**.

## Separate reshape compatibility from axis permutation

The chapter uses one shared notation-only formula:

$$
\prod_k n_k=\prod_j n'_j, \quad s'_k=s_{\pi(k)}
$$

The first equality is the element-count requirement for reshape. The source
extents `n_k` and requested extents `n'_j` must have equal checked products. For
the fixture, `2×3 = 3×2 = 6`, so `[2,3] → [3,2]` is size-compatible. Shape
`[4,2]` needs eight values and fails before any view is created.

Equal products are not the whole rule in this course. Reshape also requires the
source view to be row-major contiguous. That policy prevents `reshape` from
silently allocating after a transpose or inner-axis slice. Call `materialize`
when a new contiguous owner is intentionally required.

The second equality is for permutation. Output axis `k` takes the stride of
source axis `π(k)`, and its matching extent moves with it. For permutation
`[1,0]`, output axis `0` takes source axis `1`, while output axis `1` takes source
axis `0`; shape and strides therefore become `[3,2]` and `[1,3]`.

{/* chapter-section:symbol-glossary */}

## Account for every extent, axis, and stride

| Symbol | Operational meaning |
| :--- | :--- |
| $n_k$ | The extent of source axis $k$. In the fixture, the source extents are $n_0=2$ and $n_1=3$. |
| $n'_j$ | The requested reshape extent on axis $j$. For `[3,2]`, $n'_0=3$ and $n'_1=2$. |
| $k$ | A zero-based source or output axis index, according to the expression containing it. |
| $j$ | A zero-based requested-reshape axis index. It need not run over the same number of axes as $k$. |
| $s'_k$ | The element stride of output axis $k$ after a permutation. |
| $\pi(k)$ | The source axis placed at output axis $k$. A valid permutation names each source axis exactly once. |
| $s_{\pi(k)}$ | The source stride carried with axis $\pi(k)$ into output position $k$. |

Reshape and permutation use different halves of the formula. A contiguous
reshape derives fresh row-major strides from the requested shape. A permutation
reorders existing shape entries and strides together. Unit-step slicing does
neither: it retains the stride list, changes one extent, and adds
`start × stride[axis]` to the base offset.

Scalar shape `[]` has an empty product of one, one value, and an empty
permutation. Any zero extent gives a product of zero. The Rust implementation
still checks the complete requested suffix-stride chain, so a later enormous
extent cannot hide overflow behind an earlier zero.

## Make the copy boundary explicit without inventing a linear history

An eager transpose is easy to explain: allocate a destination, visit source
columns and rows, and push every value in transposed order. It produces an
independent buffer, which is sometimes exactly what a caller needs. It also pays
for a copy even when a read-only alternative would suffice.

The Rust historical contrast performs that eager loop:

Metadata-only rearrangement has deep roots. In
[J. K. Iliffe's 1961 description of the Genie system](https://archive.computerhistory.org/resources/access/text/2015/09/102726217-05-01-acc.pdf),
pages 25–26 explain codewords that controlled array selection. Iliffe identifies
row interchange and transposition as virtual operations that could leave stored
elements in place. The paper also identifies codeword space and extra indirect
addressing time as costs. Those codewords are not modern strides, and this course
does not claim that NumPy or Rust descended directly from Genie.

The official [NumPy copies and views guide](https://numpy.org/doc/stable/user/basics.copies.html)
describes a present-day split between a data buffer and metadata such as shape
and strides. It also warns that some reshapes of non-contiguous arrays require a
copy. NumPy may copy in those cases; this teaching API instead returns an error
until the learner asks for `materialize`. The
[NumPy internals guide](https://numpy.org/doc/stable/dev/internals.html)
corroborates that shape, start offset, and per-axis strides can reinterpret one
buffer. The official [Rust slice reference](https://doc.rust-lang.org/stable/reference/types/slice.html)
calls a slice a view into a sequence. Here, a `TensorView` extends that borrowed
idea with private n-dimensional metadata and checked transforms.

## Borrow the owner, transform metadata, copy only on command

The cumulative `Tensor` still owns all values. `Tensor::view` stores an immutable
reference to that owner, so Rust encodes the lifetime and alias rule in the type.
Shape, strides, base offset, and length are copied metadata; scalar values are
not copied.

The compile-fail example in the source is important evidence. If code creates a
view, mutates its owner, and then uses the view again, Rust rejects the conflicting
borrow. Multiple immutable views are safe. `materialize` creates an independent
owner, after which either tensor may be mutated separately.

Axis transforms preserve source identity. `reshape` checks requested-layout
overflow first, element count second, and current contiguity third. `transpose`
builds a complete axis list and swaps two entries. `permute` requires exactly one
in-range occurrence of every source axis.

## See shared storage and copied storage as different states

The diagram aligns five Rust records. The base, reshape, transpose, and slice all
name storage owner `base`; their solid shared-state treatment and hollow diamond
make that relationship visible without color. The materialized view names a new
owner and uses a double border plus filled diamond. Rejections use a dashed
border and multiplication sign.

Reshape and transpose sit side by side because equal shape `[3,2]` is the tempting
misconception. Read their stride and offset rows before their value rows. The
slice/materialization table then maps source offsets `[1,2,4,5]` to new offsets
`[0,1,2,3]`, making the copy boundary observable.

## Predict metadata and ownership before running Rust

Use the frozen owner unless a question supplies another shape.

1. Reshape `[2,3]` to `[3,2]`. Write shape, strides, logical offsets, and values.
2. Transpose axes `0` and `1`. Write the same four records. Why is this not the
   reshape result?
3. Start with shape `[2,3,4]` and strides `[12,4,1]`. Predict shape and strides
   after `permute([2,0,1])`.
4. For `slice(1, 1..3)`, write the new base offset, shape, strides, offsets, and
   values. Is it row-major contiguous?
5. Can that slice reshape directly to `[4]`? Give the safe two-step correction.
6. Write the materialized slice's owner, shape, strides, offsets, and values.
7. Explain why changing `tensor.as_mut_slice()[0]` while a subsequently used
   borrowed view exists is rejected by Rust.

<details>
  <summary>Check the seven shape, stride, order, and ownership predictions</summary>

1. Reshape gives shape `[3,2]`, strides `[2,1]`, offsets `[0,1,2,3,4,5]`, and
   values `[10,11,12,20,21,22]`. It shares the base owner.
2. Transpose gives shape `[3,2]`, strides `[1,3]`, offsets `[0,3,1,4,2,5]`, and
   values `[10,20,11,21,12,22]`. It moves each extent with its corresponding
   stride, whereas reshape preserves the contiguous linear order.
3. Permutation `[2,0,1]` maps source axes `2`, `0`, and `1` to the three output
   positions. The shape is `[4,2,3]` and the strides are `[1,12,4]`.
4. The slice starts at `0 + 1×1 = 1`. It has shape `[2,2]`, strides `[3,1]`,
   offsets `[1,2,4,5]`, and values `[11,12,21,22]`. The gap from offset `2` to
   `4` makes it non-contiguous in logical row-major order.
5. No. Equal element counts do not overcome non-contiguity under this API.
   Materialize the slice first, then reshape the new contiguous owner to `[4]`.
6. Materialization creates a new owner with shape `[2,2]`, strides `[2,1]`,
   offsets `[0,1,2,3]`, and storage `[11,12,21,22]`. The values equal the slice's
   logical values, but their addresses and owner differ.
7. `TensorView` contains an immutable borrow of the owner. Rust will not grant a
   conflicting mutable borrow while that immutable borrow is later used. This is
   a compile-time alias guarantee, not a runtime copy.

</details>

**Misconception check:** a transpose does not rearrange the owner's buffer. It
rearranges axis interpretation. Materialization is the operation that creates a
new buffer in the view's logical order.

## Carry explicit axes into broadcasting and reductions

The course now has two complementary tensor types. `Tensor` owns checked,
contiguous values. `TensorView` safely borrows those values with its own shape,
strides, and base offset. Reshape, transpose, permutation, and slicing can prepare
data for later numerical operations without hiding allocation.

This distinction matters throughout the decoder. Batched token states, attention
heads, and matrix operands will repeatedly change axis interpretation. A checked
view can express those transformations cheaply; explicit materialization keeps
the moments that allocate visible and testable.

Chapter 10 will use the visible axes to align shapes for broadcasting and will
name which axes reductions collapse. It will not infer semantic alignment merely
because two views happen to touch the same storage offsets.
