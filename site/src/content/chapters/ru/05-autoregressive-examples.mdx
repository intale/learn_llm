## Сначала найдите все три пары «вход — цель»

Авторегрессионная языковая модель предсказывает следующий токен по предшествующим
токенам. В этой главе мы построим для такого обучения пары «вход — цель», не
склеивая разные документы. Эти пары показывают, что нужно предсказывать, но не
ограничивают доступ модели к будущим позициям — это должна делать сама модель.

В главе 4 для каждого документа была получена отдельная последовательность с BOS
и EOS. Возьмём короткий пример:

```text
позиция   0   1   2   3   4   5
токен     0  41  42  43  44   1
          ^                   ^
         BOS                 EOS
```

Пусть длина контекста `T=3`, а шаг `S=1`. Для одной пары нужны
четыре исходных токена: три войдут во вход, а сдвиг целевой последовательности
потребует ещё один. Прежде чем читать дальше, выпишите входы и цели для начальных
позиций 0, 1 и 2.

Для пары с началом в позиции 0 берём исходные токены с индексами 0–3. Токены с
индексами 0–2 образуют вход, а токены с индексами 1–3 — цель:

```text
позиция 0   вход  [0, 41, 42]
            цель     [41, 42, 43]
```

Сдвигая начало на одну позицию, получаем три пары:

| Начальная позиция | Вход `x` | Цель `y` |
| ---: | :--- | :--- |
| `0` | `[0,41,42]` | `[41,42,43]` |
| `1` | `[41,42,43]` | `[42,43,44]` |
| `2` | `[42,43,44]` | `[43,44,1]` |

Каждая строка — одна авторегрессионная пара. Вход и цель вместе образуют один
авторегрессионный пример; в Rust он представлен структурой `CausalWindow`.

Последняя цель содержит EOS — так и должно быть: после токена `44` модель должна
предсказать конец документа. В следующей начальной позиции (`s=3`) доступен
только остаток `[43,44,1]`. Для пары нужны четыре исходных токена, поэтому
четвёртая пара не строится; мы не добавляем фиктивные токены.

То, что из позиции 3 нельзя построить новую пару, не означает, что токены 43, 44
и EOS пропали. Все они уже входят в полные пары выше. Оставшихся токенов просто
недостаточно для начала ещё одной пары.

## Запишите сдвиг на один токен через срезы

Для одного закодированного документа `z` и допустимой начальной позиции `s`:

$$
x^{(s)}=z_{s:s+T}, \quad y^{(s)}=z_{s+1:s+T+1}
$$

В Rust правая граница диапазона не включается, поэтому в каждом срезе ровно `T`
ID. Целевая последовательность начинается на одну позицию позже: `y_i` — токен
документа, непосредственно следующий за соответствующим `x_i`. Позиция допустима тогда и только тогда,
когда `s+T < |z|`, то есть от неё в документе есть все `T+1` исходных токенов.

Шаг выбирает начальные позиции `s=kS`, где `k=0,1,2,...`. Если в этом примере
взять `S=2`, будут рассмотрены позиции 0, 2, а затем 4. Позиции 0 и 2 дают пары;
позиция 1 пропущена из-за шага, а в позиции 4 остаётся лишь `[44,1]` — слишком
мало токенов для новой пары.

`T` задаёт длину входа и цели, а также наибольший доступный контекст внутри одной
пары. При предсказании `y_i` модель должна использовать только `x_0,...,x_i`, но
не более поздние элементы входа. `S` определяет расстояние между началами соседних
пар:
при `S=1` рассматривается каждая возможная начальная позиция, поэтому некоторые
переходы повторяются в перекрывающихся парах. Чем больше `S`, тем меньше пар;
некоторые позиции, из которых хватило бы токенов на полную пару, пропускаются, а
при `S>T` между парами могут возникнуть промежутки. Если обрабатывать каждый
документ отдельно, модель не увидит выдуманного перехода между независимыми
текстами. Раздельное хранение выборок сохраняет исходное разбиение, поэтому
последующий код может явно выбрать только обучающие документы.

Сдвиг задаёт правильные цели, но сам по себе не ограничивает доступ модели к
более поздним позициям входа. В рекуррентной модели, обрабатывающей токены слева
направо, это ограничение обеспечивает последовательное обновление скрытого
состояния. Декодеру с самовниманием нужна явная каузальная маска, закрывающая
последующие позиции.

## Что означает каждый символ

| Символ | Значение в этой главе |
| :--- | :--- |
| $z$ | ровно один документ `[BOS, токены содержимого, EOS]` |
| $T$ | положительная длина контекста; вход и цель содержат по `T` ID |
| $S$ | положительное расстояние между рассматриваемыми начальными позициями |
| $s$ | начальная позиция `kS`; пара строится, только если от неё остаётся `T+1` токенов |
| $x^{(s)}$ | входной срез, начинающийся в `s` |
| $y^{(s)}$ | целевой срез, начинающийся в `s+1` |

Правило применяется к каждому документу отдельно: документы одной выборки не
склеиваются в общую последовательность `z`. Короткий документ может не дать ни
одной пары; тогда весь документ образует остаток, из которого нельзя построить
пару. Даже при пустом содержимом последовательность имеет вид `[BOS, EOS]`: при
`T=1` она даёт корректную пару `[BOS] -> [EOS]`.

## Получайте цели из порядка токенов, а не из ручной разметки

В задаче классификации ответ задают отдельно от входа. В демонстрации на Rust у
двух последовательностей есть заданные людьми метки тональности. Такая метка не
является следующим токеном:

  caption="Две последовательности токенов с отдельно заданными метками тональности"
  label="Фрагмент кода Rust: две последовательности токенов с метками тональности, заданными отдельно от входа"
/>

Для языковой модели правильные ответы уже содержатся в самой последовательности:
после каждого контекста, за которым в документе есть ещё один токен, этот токен и
служит целью. [Бенжио и соавторы (2003)](https://www.jmlr.org/papers/v3/bengio03a.html)
раскладывают вероятность предложения в произведение условных вероятностей
следующего слова при фиксированном числе предшествующих слов. В [отчёте GPT-2 (2019)](https://cdn.openai.com/better-language-models/language-models.pdf)
используется та же авторегрессионная факторизация; авторы сопоставляют обучение на
текстах без специальной разметки с обучением на наборах, размеченных под
конкретные задачи.

Это сравнение не означает, что для ранних языковых моделей люди вручную размечали
каждое следующее слово. Эти работы также не определяют выбранные для курса
разбиение 8/2/2, границы документов, шаг и правило обработки остатка, из которого
нельзя построить пару. В нашей реализации каждый документ хранится вместе со
своим ID и принадлежностью к выборке. Поэтому код обучения может явно использовать
только обучающие документы, а число примеров остаётся воспроизводимым.

## Стройте полные пары отдельно для каждого документа

`CausalWindowConfig::new` отклоняет нулевую длину контекста или нулевой шаг. Он
также отклоняет `T=usize::MAX`, потому что значение `T+1` вызвало бы переполнение.
При допустимых параметрах конфигурация подсчитывает число выбранных шагом
начальных позиций, из которых можно построить полный участок длиной `T+1`, и
находит остаток в следующей позиции, если токенов в нём уже недостаточно для новой
пары:

По одним ID токенов итератор не может определить границы документов, поэтому ему
нужно передавать срез ровно одного документа. Основной путь создаёт отдельный
итератор для каждого `EncodedDocument`.

Метод `incomplete_tail` возвращает остаток, начинающийся в первой выбранной шагом
позиции, из которой уже нельзя построить пару. При вычислении этой позиции
проверяется переполнение. Если позиция находится в конце документа или за ним,
метод возвращает `None`. Токены найденного остатка могли уже войти в предыдущие
полные пары.

`EncodedCorpusPartitions` хранит закодированные последовательности вместе с ID
документов и принадлежностью к выборке. Метод `from_partitions` независимо
кодирует каждый документ токенизатором, зафиксированным в главе 4; метод
`documents` возвращает документы только из явно указанной выборки. Здесь
используются окончательные ID токенизатора из главы 4, а не вспомогательное
пространство ID, в котором в главе 3 обучались правила слияния. Корпус не
объединяется в одну последовательность:

## Определите начальные позиции и сверьте ответы

1. Перечислите все входы и цели для `[0,41,42,43,44,1]` при `T=3, S=1`.
2. Повторите вычисление при `S=2`. Какие начальные позиции дают пары? Из какой пропущенной позиции тоже можно было бы построить полную пару? Какие токены остаются в следующей позиции, выбранной шагом?
3. Для `[0,61,1]` и `T=3` предскажите число пар и оставшиеся токены. Почему добавление фиктивных токенов означало бы другое правило подготовки целей?
4. Получится ли при `T=1` из `[0,1]` пара `[BOS] -> [EOS]`?
5. Даны документы `[0,10,1]` и `[0,20,1]`. Может ли при `T=2` появиться пара `[10,1] -> [1,0]`?
6. При `T=3, S=5` токенов в документе хватает ровно на одну полную пару. Нужно ли сообщать о пустом остатке?
7. Объясните, как токены из остатка, которого недостаточно для новой пары, могли уже попасть в предыдущие пары.
8. Объясните, почему в главе 6 нельзя подсчитывать переходы обходом всех перекрывающихся пар.
9. Мешает ли сдвиг целевой последовательности модели читать более поздние позиции входа?

замените только шаг в `CausalWindowConfig::new(3, 1)` в файле

выполните `cargo run --quiet --locked -p ch05-autoregressive-examples`. Другой

корпусе тоже изменится, но вычислять его вручную не нужно. Сверьте с прогнозом
результат для короткой последовательности, а затем восстановите

исходная конфигурация.

**Распространённое заблуждение:** «Остаток, из которого нельзя построить новую
пару, отбрасывается». Нет. Новая пара из него не начинается, но те же токены могли
уже войти в полные пары с более ранними начальными позициями.

## Свяжите пары с задачей декодера

Каждая построенная пара показывает, чему декодер должен научиться в каждой
позиции: по текущему токену и доступному предшествующему контексту предсказать
следующий токен того же документа. BOS может быть первым входным токеном, а EOS —
последней целью. BOS и EOS обозначают границы документа и не используются для
заполнения последовательности.

Сдвинутые последовательности показывают, что нужно предсказывать в каждой
позиции, но не мешают модели видеть более поздние элементы входа. В рекуррентной
модели, обрабатывающей токены слева направо, это ограничение обеспечивает
последовательное обновление скрытого состояния. Декодеру с самовниманием нужна
каузальная маска, закрывающая последующие позиции.

В следующих главах мы перейдём от целочисленного подсчёта переходов к тензорным
вычислениям, эмбеддингам, вниманию и обучаемым параметрам, из которых модель
получает логиты. Но связь между входом и целью останется прежней.
Глава 6 сначала построит наглядную биграммную модель. Она прочитает исходные
закодированные документы **обучающей выборки**, а не перекрывающиеся пары, чтобы
каждый соседний переход учитывался ровно один раз, а валидационные и тестовые
данные не влияли на параметры.
