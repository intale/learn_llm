## Предскажите первое слияние до запуска программы

Возьмём пробел и следующую за ним кириллическую `а`:

```text
текст         = " а"
байты UTF-8   = [20, d0, b0]
ID обучения   = [32, 208, 176]
```

Из главы 3 нужны два правила:

| Ранг | Пара в пространстве обучения | Новый ID обучения | Сохранённые байты |
| ---: | :--- | ---: | :--- |
| 0 | `(32,208)` | `256` | `20 d0` |
| 1 | `(208,176)` | `257` | `d0 b0` |

Какое правило сработает? Ранг задаёт приоритет, а не частоту, которую надо заново
посчитать для этого входа. Сначала ранг 0 поглощает `32,208`, и остаётся
`[256,176]`. Ранг 1 уже не видит рядом `208,176`. После резервирования двух
управляющих токенов каждый ID обучения сдвигается на два:

```text
канонические ID содержимого = [258, 178]
ID документа                = [0, 258, 178, 1]
                               ^             ^
                              BOS           EOS
```

Теперь восстановим содержимое. В токене `258` хранятся байты `20 d0`, а токен
`178` представляет исходный байт `b0`. Их конкатенация даёт `20 d0 b0` — точный
вход. Сам по себе токен `258` не является корректным UTF-8: последний байт `d0`
начинает кодировку, а байт продолжения приходит из следующего токена. Токены —
это части последовательности байтов, а не обязательно символы.

Есть важная ловушка. `[34,259]` — тоже допустимая последовательность содержимого:
в токене `34` хранится пробел `20`, а в токене ранга 1 с ID `259` — `d0 b0`.
Декодирование даёт те же три байта. Но повторное кодирование вернёт канонические
`[258,178]`, потому что ранг 0 имеет приоритет. Точное декодирование не означает,
что любая допустимая последовательность токенов канонична.

## Гарантируйте точные байты в одном направлении

Токенизатор гарантирует:

$$
\operatorname{decode}_{content}(\operatorname{encode}_{content}(x))=\operatorname{bytes}(x)
$$

`encode_content` начинает с одного сдвинутого ID на байт и применяет
зафиксированные правила от ранга 0 по возрастанию. Функция не смотрит на частоты
валидационной или тестовой выборки, не обучает новые пары, не нормализует Unicode,
не меняет регистр и не делит строку на слова. `decode_content` находит сохранённую
байтовую развёртку каждого токена содержимого и объединяет развёртки.

Формула намеренно не утверждает, что
`encode_content(decode_content(z)) = z` для любой последовательности `z`.
Контрпример — `[34,259]`. Модели достаточно именно записанной гарантии: один
детерминированный кодировщик и декодировщик без потери байтов для его результатов.

## Обозначения и схема ID

| Обозначение | Смысл |
| :--- | :--- |
| $x$ | входное содержимое, переданное как текст UTF-8 или непосредственно как байты |
| $\operatorname{encode}_{content}$ | преобразование байтов в исходные ID и последующее применение всех зафиксированных рангов по возрастанию, без BOS и EOS |
| $\operatorname{decode}_{content}$ | конкатенация сохранённых байтовых развёрток всех ID токенов содержимого |
| $\operatorname{bytes}(x)$ | точная входная последовательность байтов без нормализации Unicode и подстановок |

В версии 1 используется единое непрерывное пространство ID:

| ID | Значение | Отображение |
| :--- | :--- | :--- |
| `0` | управляющий токен BOS | фиксирован |
| `1` | управляющий токен EOS | фиксирован |
| `2..=257` | 256 однобайтовых токенов содержимого | байт `b` переходит в `b+2` |
| `258..` | обученные токены содержимого | ранг `r` переходит в `258+r` |

В курсе нет ID для PAD. Далее будут использоваться окна фиксированной длины, а
обслуживание с большим объёмом дополнения отложено. Это граница курса, а не
утверждение, будто рабочим системам никогда не требуется padding.

## Решите проблему неизвестных строк с помощью байтового покрытия

Закрытый словарь целых слов может дать по одному ID частым словам из обучающей
таблицы, но для невиданного написания строки в таблице нет. Исторический пример
на Rust отображает `lowering` и любое другое отсутствующее слово в ID `0`;
декодер восстанавливает лишь маркер `<UNK>`, а не исходное написание:

<RustSource
  path="rust/demos/ch04-apply-bpe-tokenizer/src/lib.rs"
  region="unknown-token-loss"
  caption="Закрытая таблица слов, которая не может восстановить невиданное написание"
  label="Функции Rust, отображающие невиданные целые слова в один неизвестный ID и восстанавливающие только маркер неизвестного токена"
/>

В [работе 2016 года о редких словах](https://aclanthology.org/P16-1162/)
Сеннич, Хэддоу и Бёрч перенесли повторное слияние пар на подслова из
последовательностей символов. Так получаются полезные единицы между целыми словами и
символами, однако эта статья не служит доказательством универсальной байтовой базы.

В [разделе о представлении входа отчёта GPT-2](https://cdn.openai.com/better-language-models/language-models.pdf)
объясняется следующий шаг: для байтов UTF-8 достаточно базового алфавита из 256
символов, который может представить любую строку Unicode. За покрытие приходится
платить длиной. Кириллическая буква занимает два байта UTF-8 до слияний, emoji —
часто четыре. Обученный словарь может сократить частые последовательности, но
байтовое покрытие не гарантирует ни минимальный конечный словарь, ни одинаково
короткие последовательности для разных письменностей.

В GPT-2 также применялись ограничения по категориям символов с исключением для
пробела. Наш токенизатор устроен иначе: он использует точные правила главы 3,
разрешает любую пару внутри одного документа и останавливается только на границе
документа. Явное перечисление отличий не позволяет слову «BPE» скрыть важные
политики реализации.

## Соберите зафиксированный строгий токенизатор на Rust

Схема ID делает разделение управляющих токенов и содержимого исполняемым
инвариантом, а не комментарием. Код проверяет, что последний ID помещается в
`u32`, и открывает диапазоны байтов и слияний:

  region="token-id-layout"
  caption="Версия 1 схемы токенизатора и проверяемые диапазоны ID"

`BpeTokenizer::from_training` копирует упорядоченную таблицу пар и заново строит
все байтовые развёртки, поэтому готовый токенизатор не зависит от времени жизни
объекта обучения. Проверяемый конструктор из таблицы пар даёт упражнениям — и
будущим контрольным точкам — тот же инвариант: на ранге `r` оба операнда уже
должны существовать, а одна пара не может повторяться.

Основной вход кодировщика имеет тип `&[u8]`. Вариант для `&str` лишь передаёт его
байты UTF-8. Каждое правило сдвигается на `+2` и один раз применяется в порядке
рангов с заменой слева направо из главы 3. Необязательная трассировка записывает
только те ранги, которые изменили конкретный вход, но не влияет на результат:

## Сначала предскажите, затем проверьте

1. Для байтов `20 d0 b0` предскажите результат при ошибочном применении ранга 1
   до ранга 0, а затем запишите канонический результат.
2. Переведите ASCII `?` (байт `3f`), ID слияния `257` из пространства обучения
   и ранг `7` в схему версии 1.
3. Предскажите `encode_content("")` и `encode_document("")`.
4. Предскажите ID содержимого невиданного `🦀`, если ни одна обученная пара не
   совпадает.
5. Кириллическая `т` кодируется в UTF-8 как `d1 82`, а ранг 2 выучил
   `(209,130)`. Предскажите конечную последовательность.
6. Декодируйте `[34,259]`, затем предскажите результат повторного кодирования.
7. Найдите первую ошибку в документе `[0,100,0,1]` и содержимом `[100,1]`.
8. Решите, обязано ли точное восстановление байтов из ID `[257,256]` давать корректный
   `String`.

<details>
  <summary>Проверить ответы</summary>

  1. Неверный порядок создал бы `[32,257]` в пространстве обучения и `[34,259]`
     после сдвига. Правильный порядок даёт `[256,176]` и `[258,178]`.
  2. `3f` — это 63, поэтому `?` получает ID `65`. ID обучения `257` переходит в
     `259`, а ранг 7 — в `258+7=265`.
  3. Пустое содержимое — `[]`, пустой документ — `[0,1]`. Оба декодируются в
     ноль байтов.
  4. `🦀` имеет байты `f0 9f a6 80`, поэтому резервные ID после сдвига равны
     `[242,161,168,130]`.
  5. Сдвинутые операнды `(211,132)` сливаются на ранге 2 в ID `260`, поэтому
     результат — `[260]`.
  6. `[34,259]` раскрывается в `20 d0 b0`, то есть в `" а"`. При повторном
     кодировании сначала срабатывает ранг 0, и получается `[258,178]`.
  7. BOS в позиции 2 документа является внутренним управляющим токеном. EOS в
     позиции 1 содержимого запрещён: `decode_content` не принимает управляющие ID.
  8. Нет. ID `[257,256]` точно восстанавливают `ff fe`, а строгое преобразование
     в UTF-8 отвергает результат на байте 0. Главный контракт — обратимость байтов.
</details>

**Проверка заблуждения:** «Если декодирование не теряет данные, любая допустимая
последовательность токенов является результатом кодировщика для своих байтов».
Это неверно. Несколько разбиений могут декодироваться без потерь, а порядок
зафиксированных рангов выбирает одно каноническое кодирование.

## Сохраняйте отдельную последовательность с границами для каждого документа

Теперь накопительная реализация преобразует каждый исходный документ в одну
детерминированную последовательность `[BOS, содержимое..., EOS]` и восстанавливает
исходные байты. Статистика слияний по-прежнему приходит только из обучающих
документов главы 2; применение зафиксированного токенизатора к валидационному или
тестовому содержимому не переобучает его.

Глава 5 получит эти последовательности по отдельности. Она построит сдвинутые
окна контекста и цели только внутри одной последовательности, не объединяя
документы и не пересекая границы обучающей, валидационной и тестовой выборок.
BOS задаст начальный контекст, а EOS станет последней целью предсказания. Оба
токена сохранят явный смысл границы документа и не будут участвовать в слияниях.
