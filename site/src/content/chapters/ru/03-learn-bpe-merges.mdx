## Предскажите два раунда до запуска программы

Рассматривайте эти строки как **два документа**, а не как одну объединённую строку:

```text
train-aaa = aaa = [97, 97, 97]
train-aba = aba = [97, 98, 97]
```

Здесь `97` — байт `61` в шестнадцатеричной записи (`a`), а `98` — байт `62`
(`b`). Прежде чем читать дальше, перечислите все соседние пары. В `aaa` пара
`(97,97)` начинается в позициях 0 и 1. Оба вхождения включают средний байт, но при
подсчёте пар-кандидатов всё равно учитываются отдельно. Второй документ добавляет
по одному вхождению `(97,98)` и `(98,97)`:

| Пара | Позиции в двух документах | Частота |
| :--- | :--- | ---: |
| `(97,97)` | `train-aaa: 0–1, 1–2` | 2 |
| `(97,98)` | `train-aba: 0–1` | 1 |
| `(98,97)` | `train-aba: 1–2` | 1 |

Поэтому на ранге 0 выбирается `(97,97)`, и этой паре назначается новый ID `256`.
Теперь выполните один проход замены слева направо. Токены в позициях 0–1
заменяются одним токеном `256`, после чего просмотр продолжается с позиции 2.
Средний `97` нельзя повторно использовать ещё в одной замене. Получаются такие
последовательности:

```text
train-aaa = [256, 97]
train-aba = [97, 98, 97]
```

Это главное различие главы: **частота пары-кандидата равна 2**, а **число замен
равно 1**. По частотам выбирают правило, а проход замены применяет его к
последовательности, не используя один токен дважды.

Теперь предскажите результат на ранге 1. Кандидаты `(97,98)`, `(98,97)` и
`(256,97)` встречаются по одному разу. В курсе сравниваются числовые пары, поэтому
побеждает `(97,98)`: её левый ID `97` меньше `98` и `256`. Новому ID `257`
соответствуют байты `61 62`, и получается:

```text
train-aaa = [256, 97]
train-aba = [257, 97]
```

Наконец, рассмотрите два однобайтовых документа `a | a`. Черта обозначает границу
документов и не входит ни в один из них. Длина каждой последовательности равна 1,
поэтому в ней нет окна длины 2. Таблица частот кандидатов пуста: искусственная пара
`(97,97)` через границу не возникает.

## Задайте однозначное правило выбора

В каждом раунде выбранная пара и новый токен определяются так:

$$
(a^{*},b^{*})=\arg\max_{(a,b)}\bigl(C(a,b),-a,-b\bigr),\quad m^{*}=a^{*}\Vert b^{*}
$$

Трёхкомпонентные кортежи сравниваются лексикографически. Сначала максимизируется
частота `C(a,b)`. При равных частотах компонент `-a` отдаёт предпочтение меньшему
левому ID; если левые ID тоже равны, компонент `-b` выбирает меньший правый ID.
Такое числовое правило делает результат курса воспроизводимым, но не описывает все
реализации BPE.

Во втором выражении объединяются **последовательности байтов токенов**. Если токену
`256` соответствуют байты `61 61`, а токену `97` — байт `61`, то для результата
слияния `(256,97)` сохраняются байты `61 61 61`. Десятичные строки `"256"` и `"97"`
не склеиваются. Если ни в одном документе нет соседней пары, обучение завершается
раньше; в остальных случаях правило можно создать и для пары с частотой 1.

## Обозначения

| Обозначение | Значение |
| :--- | :--- |
| $a,b$ | числовые ID двух соседних токенов в текущем раунде — левого и правого |
| $(a,b)$ | упорядоченная соседняя пара; если поменять ID местами, получится другая пара-кандидат |
| $C(a,b)$ | число вхождений этой пары в соседних позициях всех обучающих документов; перекрытия учитываются, а границы документов — нет |
| $\arg\max$ | выбор кандидата с лексикографически наибольшей трёхкомпонентной оценкой |
| $-a,-b$ | детерминированное правило курса: при равных частотах выбирается пара с меньшим левым ID, а при его равенстве — с меньшим правым ID |
| $a^{*},b^{*}$ | выбранные левый и правый ID; звёздочка отмечает победителя и не означает умножение |
| $m^{*}$ | новый токен в пространстве ID обучения; его ID равен 256 плюс ранг при нумерации с нуля |
| $\Vert$ | объединение последовательностей байтов, соответствующих двум ID, а не арифметическая операция над ID |

Исходные байты занимают ID `0..=255`. Токен, созданный на успешном раунде `r` при
нумерации с нуля, получает ID `256+r`, поэтому после `k` успешных раундов
пространство ID обучения содержит `256+k` токенов. В главе 4 эти ID будут
преобразованы в окончательную схему после резервирования управляющих токенов.

## От закрытой таблицы слов к многократному слиянию пар

Фиксированный словарь целых слов назначает ID каждому написанию, встречавшемуся при
обучении. Такая схема проста, но не умеет различать новые написания. В исполняемом
примере слово `lower` имеет собственный ID, а `lowering` и любое другое слово,
отсутствующее в словаре, получают один и тот же ID `0`:

<RustSource
  path="rust/demos/ch03-learn-bpe-merges/src/lib.rs"
  region="whole-word-unknown"
  caption="Детерминированный словарь целых слов с одним ID неизвестного токена"
  label="Функции на Rust, назначающие ID целым словам и сопоставляющие каждому отсутствующему в словаре слову ID 0"
/>

В алгоритме сжатия Филипа Гейджа 1994 года применялся другой подход: найти частую
соседнюю пару байтов, заменить её неиспользуемым значением байта и сохранить
таблицу замен, достаточную для обратного преобразования. Изучите
[исходное описание сжатия парами байтов](https://jacobfilipp.com/DrDobbs/articles/CUJ/1994/9402/gage/gage.htm).
Этот алгоритм решает другую задачу и ограничен доступными значениями байтов. В
нашем токенизаторе ID имеют тип `u32`, а для каждого токена хранится соответствующая
последовательность байтов.

Позже Рико Сеннич, Барри Хэддоу и Александра Бёрч применили повторное слияние пар
в нейронном машинном переводе с открытым словарём. В методе из их
[статьи 2016 года о подсловах](https://aclanthology.org/P16-1162/) исходными
единицами служат последовательности символов, а слияния через границы слов
запрещены. В этом курсе намеренно выбран другой вариант: исходными единицами служат
байты UTF-8, пробел может входить в пару внутри документа, а слияния останавливаются
только на границе документа.

Эти различия существенны. Название BPE относится к семейству алгоритмов повторного
слияния пар и не задаёт единственного правила выбора при равных частотах или
обработки границ. Принятое в курсе правило выбора численно наименьшей пары явно
сформулировано и проверено тестами, чтобы один и тот же корпус всегда давал
одинаковые ранги.

## Реализуйте обучение без готовой библиотеки токенизации

Счётчик пар принимает срез последовательностей, а не один плоский массив.
`windows(2)` вызывается отдельно для каждого документа: так учитываются оба
перекрывающихся окна в `aaa`, а окно не может пересечь границу документа.
`BTreeMap` хранит кандидатов в лексикографическом порядке числовых пар. При обходе
победитель меняется только при строго большей частоте, поэтому при равенстве
остаётся первая, то есть наименьшая, пара:

UTF-8, кодирующих кириллические символы, и без байта продолжения остаётся
незавершённым. Поэтому `20 d0` — **не отдельный символ и не корректная
самостоятельная строка UTF-8**. Эту последовательность следует хранить и показывать
как байты или шестнадцатеричные значения.

При сборке схема разбирает тот же эталонный вывод программы на Rust. В каждом
состоянии видны оба документа, а таблица кандидатов отдельно показывает частоту с
учётом перекрытий и число замен без перекрытий. Выбранная строка отмечена текстом,
звёздочкой, жирным начертанием и рамкой, поэтому цвет не обязателен. Числовые и
шестнадцатеричные фрагменты сохраняют направление слева направо при любом языке
интерфейса, а на узком экране блоки располагаются друг под другом без клиентского
скрипта.

Ещё раз найдите в состоянии 0 средний токен `97` документа `aaa`: он входит в оба
перекрывающихся кандидата. Сопоставьте частоту `2` и число замен `1` на ранге 0. На
ранге 1 не опирайтесь на положение строк: примените правило сравнения числовых
кортежей и выберите `(97,98)` вместо `(98,97)` и `(256,97)`.

## Сначала предскажите, затем проверьте

1. Для одного документа `[97,97,97,97]` вычислите `C(97,97)` и число замен за
   один проход слева направо.
2. После ранга 0 в учебном примере выпишите кортеж оценки для каждого из трёх
   кандидатов с частотой 1 и выберите победителя.
3. Объясните, почему документы `[97] | [98]` не создают кандидата `(97,98)`.
4. Предскажите размер словаря после 12 запрошенных раундов, если обучение
   остановилось после 9 успешных.
5. Определите, могут ли 10 000 повторов в валидационном документе изменить таблицу
   рангов.
6. Запишите последовательность байтов гипотетического результата слияния
   `(256,97)`, если токену `256` соответствуют байты `61 61`.
7. Объясните, почему нельзя печатать байты реального ранга 0 `20 d0` как символ.

<details>
  <summary>Проверьте ответы</summary>

  1. У четырёх одинаковых токенов три соседние позиции, поэтому `C(97,97)=3`.
     Один проход заменяет позиции 0–1 и 2–3: результат `[256,256]`, две замены.
  2. Оценки равны `(1,-97,-98)`, `(1,-98,-97)` и `(1,-256,-97)`.
     Лексикографический максимум выбирает `(97,98)`.
  3. Окна строятся отдельно внутри каждого документа. Обе последовательности
     имеют длину 1, поэтому ни в одной нет соседней позиции.
  4. В словаре `256+9=265` токенов. Число запрошенных раундов задаёт верхнюю
     границу; токен добавляется лишь в успешном раунде.
  5. Нет. Тренер получает статистику только из `training_documents()`;
     валидационная и тестовая выборки не участвуют в обучении.
  6. Последовательности байтов объединяются: за `61 61` следует `61`, итого
     `61 61 61`.
  7. `d0` — начальный байт UTF-8 без байта продолжения. Токен, полученный при
     обучении, представляет последовательность байтов, которая не обязана быть
     символом, словом или корректным самостоятельным текстом.
</details>

При изменении реализации сначала предскажите первый ранг, который должен
измениться. Если вывод стал другим, отдельно проверьте состав корпуса, границы
документов, подсчёт, правило выбора при равных частотах и замену. Само название BPE
не делает разные реализации взаимозаменяемыми.

## Зафиксируйте ранги до их применения

В этой главе сформирована упорядоченная таблица правил, но способ кодирования
произвольных входных данных ещё не определён. Этапы разделены намеренно: при
обучении формируется порядок слияний, а при кодировании уже зафиксированные ранги
применяются без повторного подсчёта частот на новом тексте.

В главе 4 будут зарезервированы BOS и EOS, а каждый ID содержимого из главы 3 будет
сдвинут на два. Зафиксированные ранги будут применяться к произвольному тексту в
UTF-8 или последовательности байтов, после чего декодер восстановит байты каждого
ID. Порядок важен: раннее слияние может создать токен, который станет операндом
более позднего правила. Байты валидационной и тестовой выборок не учитываются при
расчёте статистики слияний, даже когда следующая глава проверяет обратное
преобразование на отложенных примерах.
