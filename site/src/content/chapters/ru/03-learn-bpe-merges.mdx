## Предскажите два раунда до запуска программы

Считайте эти строки **двумя документами**, а не одной склеенной строкой:

```text
train-aaa = aaa = [97, 97, 97]
train-aba = aba = [97, 98, 97]
```

Здесь `97` — байт `61` в шестнадцатеричной записи (`a`), а `98` — байт `62`
(`b`). Прежде чем читать дальше, перечислите все соседние позиции. В `aaa` пара
`(97,97)` начинается в позициях 0 и 1. Эти два вхождения используют общий средний
байт, но всё равно считаются двумя позициями-кандидатами. Второй документ добавляет
по одному вхождению `(97,98)` и `(98,97)`:

| Пара | Позиции в двух документах | Частота |
| :--- | :--- | ---: |
| `(97,97)` | `train-aaa: 0–1, 1–2` | 2 |
| `(97,98)` | `train-aba: 0–1` | 1 |
| `(98,97)` | `train-aba: 1–2` | 1 |

Поэтому в ранге 0 выбирается `(97,97)`, и ей назначается новый ID `256`. Теперь
сделайте один проход замены слева направо. Совпадение в позициях 0–1 поглощает оба
входных токена, поэтому сканирование продолжается с позиции 2. Средний `97` нельзя
повторно использовать во второй замене. Получаются состояния:

```text
train-aaa = [256, 97]
train-aba = [97, 98, 97]
```

Это главное различие главы: **частота кандидата равна 2**, а **число замен равно
1**. Подсчёт определяет, какое правило выучить; замена определяет, как одно правило
преобразует последовательность, не используя один токен дважды.

Предскажите ранг 1. Кандидаты `(97,98)`, `(98,97)` и `(256,97)` встречаются по
одному разу. В курсе сравниваются числовые пары, поэтому побеждает `(97,98)`: её
левый ID `97` меньше `98` и `256`. Новый ID `257` представляет байты `61 62`, и
получается:

```text
train-aaa = [256, 97]
train-aba = [257, 97]
```

Наконец, рассмотрите два однобайтовых документа `a | a`. Черта обозначает границу
документов и не входит ни в один из них. Длина каждой последовательности равна
одному, поэтому в ней нет окна длины два. Карта кандидатов пуста: искусственная
пара `(97,97)` через границу не возникает.

## Выберите одно воспроизводимое правило

В каждом раунде победитель и новый символ определяются так:

$$
(a^{*},b^{*})=\arg\max_{(a,b)}\bigl(C(a,b),-a,-b\bigr),\quad m^{*}=a^{*}\Vert b^{*}
$$

Трёхкомпонентные кортежи сравниваются лексикографически. Сначала максимизируется
частота `C(a,b)`. При равенстве максимизация `-a` выбирает меньший левый ID; если и
они равны, максимизация `-b` выбирает меньший правый ID. Это числовое правило —
политика воспроизводимости данного курса, а не утверждение обо всех реализациях
BPE.

Второе выражение действует над **байтовыми развёртками**. Если токен `256`
раскрывается в `61 61`, а токен `97` — в `61`, то слияние `(256,97)` сохранит
`61 61 61`. Десятичные строки `"256"` и `"97"` не склеиваются. Если ни в одном
документе нет соседней пары, обучение завершается раньше; в остальных случаях
можно выучить и кандидата с частотой один.

{/* chapter-section:symbol-glossary */}

## Обозначения

| Обозначение | Значение |
| :--- | :--- |
| $a,b$ | числовые ID левого и правого соседних символов в текущем раунде |
| $(a,b)$ | упорядоченная соседняя пара; перестановка ID образует другого кандидата |
| $C(a,b)$ | число соседних позиций с этой парой во всех обучающих документах, включая перекрытия и исключая границы документов |
| $\arg\max$ | выбор кандидата с лексикографически наибольшей трёхкомпонентной оценкой |
| $-a,-b$ | детерминированное правило курса для равных частот: после частоты побеждает меньший левый, затем меньший правый числовой ID |
| $a^{*},b^{*}$ | выбранные левый и правый ID; звёздочка отмечает победителя и не означает умножение |
| $m^{*}$ | новый символ пространства обучения с ID, равным 256 плюс его ранг, начинающийся с нуля |
| $\Vert$ | конкатенация байтовых развёрток двух ID, а не арифметическая операция над ID |

Исходные байты занимают ID `0..=255`. Успешный ранг `r`, нумеруемый с нуля,
получает ID `256+r`, поэтому после `k` успешных раундов пространство обучения
содержит `256+k` символов. В главе 4 эти ID содержимого попадут в окончательную
схему после резервирования управляющих токенов.

## От закрытой таблицы слов к повторному слиянию пар

Фиксированный словарь целых слов назначает ID каждому написанию, увиденному при
обучении. Такая схема проста, но для любого нового написания нужен запасной
вариант. В исполняемом сравнении слово `lower` имеет собственный ID, тогда как
`lowering` и любое другое невиданное слово сводятся к одному ID `0`:

  caption="Детерминированная таблица целых слов с одним неизвестным ID"
  label="Функции Rust, назначающие ID целым словам и переводящие каждое невиданное слово в нулевой ID"
/>

В алгоритме сжатия Филипа Гейджа 1994 года использовался другой путь: найти частую
соседнюю пару байтов, заменить её свободным байтом и сохранить таблицу замен,
достаточную для обратного преобразования. Изучите
[исходное описание сжатия парами байтов](https://jacobfilipp.com/DrDobbs/articles/CUJ/1994/9402/gage/gage.htm).
Цель и ограниченный байтовый механизм замены отличаются от нашего токенизатора:
здесь ID имеют тип `u32`, а для каждого символа хранится его байтовая развёртка.

Позже Рико Сеннич, Барри Хэддоу и Александра Бёрч перенесли повторное слияние пар
в нейронный машинный перевод с открытым словарём. Их
[статья 2016 года о подсловах](https://aclanthology.org/P16-1162/) начинает с
последовательностей символов и запрещает слияния через границы слов. В этом курсе
намеренно выбран другой вариант: исходными символами служат байты UTF-8, пробел
может войти в пару внутри документа, а жёстким барьером является только граница
документа.

Эти различия существенны. Название BPE относится к семейству процедур повторного
слияния пар, а не к единственному правилу равных частот и границ. Наше правило
выбора численно наименьшей пары сформулировано и проверено тестами, чтобы один
корпус всегда давал одинаковые ранги.

## Реализуйте обучение без готовой библиотеки токенизации

Счётчик пар принимает срез последовательностей, а не один плоский массив. Вызов
`windows(2)` выполняется отдельно для каждого документа: он учитывает оба
перекрывающихся окна в `aaa` и не способен пересечь границу. `BTreeMap` хранит
кандидатов в числовом порядке пар; при обходе победитель меняется только при
строго большей частоте, поэтому при равенстве остаётся первая, то есть наименьшая,

## Изучите ту же трассировку на статической схеме

При сборке схема разбирает тот же эталонный вывод Rust. В каждом состоянии видны
оба документа; таблица кандидатов отделяет подсчёт с перекрытиями от числа замен

располагаются друг под другом без клиентского скрипта.

Ещё раз найдите в состоянии 0 общий средний `97` документа `aaa`. Сопоставьте
частоту `2` и число замен `1` в ранге 0. В ранге 1 не опирайтесь на положение
строк: примените правило числового кортежа, по которому `(97,98)` обязана победить
`(98,97)` и `(256,97)`.

## Сначала предскажите, затем проверьте

1. Для одного документа `[97,97,97,97]` вычислите `C(97,97)` и число замен за
   один проход слева направо.
2. После ранга 0 в учебном примере выпишите кортеж оценки для каждого из трёх
   кандидатов с частотой один и выберите победителя.
3. Объясните, почему документы `[97] | [98]` не создают кандидата `(97,98)`.
4. Предскажите размер словаря после 12 запрошенных раундов, если обучение
   остановилось после 9 успешных.
5. Решите, могут ли 10 000 повторов в валидационном документе изменить таблицу
   рангов.
6. Раскройте гипотетическое слияние `(256,97)`, если токен `256` хранит байты
   `61 61`.
7. Объясните, почему нельзя печатать байты реального ранга 0 `20 d0` как символ.

<details>
  <summary>Проверить ответы</summary>

  1. У четырёх одинаковых токенов три соседние позиции, поэтому `C(97,97)=3`.
     Один проход заменяет позиции 0–1 и 2–3: результат `[256,256]`, две замены.
  2. Оценки равны `(1,-97,-98)`, `(1,-98,-97)` и `(1,-256,-97)`.
     Лексикографический максимум выбирает `(97,98)`.
  3. Окна строятся отдельно внутри каждого документа. Обе последовательности
     имеют длину один, поэтому ни в одной нет соседней позиции.
  4. В словаре `256+9=265` символов. Число запрошенных раундов задаёт верхнюю
     границу; символ добавляется лишь в успешном раунде.
  5. Нет. Тренер получает статистику только из `training_documents()`;
     валидационная и тестовая выборки остаются отложенными.
  6. Развёртки конкатенируются: за `61 61` следует `61`, итого `61 61 61`.
  7. `d0` — начальный байт UTF-8 без байта продолжения. Выученный токен — это
     последовательность байтов, которая не обязана быть символом, словом или
     корректным самостоятельным текстом.
</details>

При изменении реализации сначала предскажите первый ранг, который должен
измениться. Если вывод стал другим, отдельно проверьте происхождение корпуса,
границы документов, подсчёт, правило равных частот и замену, а не считайте все
реализации под названием BPE взаимозаменяемыми.

## Зафиксируйте ранги до их применения

В этой главе выучена упорядоченная таблица правил, но ещё не определено кодирование
произвольного входа. Разделение намеренно: обучение обнаруживает ранги, а
кодирование позже применяет уже зафиксированные ранги, не пересчитывая частоты на
новом тексте.

В главе 4 будут зарезервированы BOS и EOS, каждый ID содержимого из главы 3 будет
сдвинут на два, ранги будут применены к произвольным байтам UTF-8, а декодер
восстановит байтовую развёртку каждого ID. Порядок важен: раннее слияние может
создать операнд для более позднего. Байты валидационной и тестовой выборок не
участвуют ни в одной статистике слияний, даже когда следующая глава проверяет
обратное преобразование на отложенных примерах.
