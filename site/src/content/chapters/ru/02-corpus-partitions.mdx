## Сначала проверьте идентификаторы, затем читайте тексты

Пусть в корпусе шесть документов со стабильными идентификаторами от `doc-01` до
`doc-06`. Прежде чем читать ответ, найдите ошибки в таком правдоподобном
разбиении:

```text
train      = [doc-01, doc-02]
validation = [doc-03, doc-04]
test       = [doc-04, doc-05]
```

Здесь нарушены два разных свойства. `doc-04` попал и в валидационную, и в
тестовую выборку, поэтому они пересекаются. `doc-06` не попал никуда, поэтому
объединение выборок не покрывает корпус. Один из вариантов исправления:

```text
train      = [doc-01, doc-02]
validation = [doc-03, doc-04]
test       = [doc-05, doc-06]
```

Теперь задайте более важный вопрос: какую исходную единицу обозначает каждый ID?
Если два идентификатора относятся к перекрывающимся отрезкам одного документа,
разные имена не мешают словам и окружающему контексту оказаться по обе стороны
границы. Поэтому сначала мы назначаем целый исходный документ. Токены и обучающие
окна можно создавать лишь после этого и только внутри выбранной роли.

В репозитории лежат двенадцать коротких документов UTF-8: английская и русская
версии шести сцен. Два перевода одной сцены образуют общую группу происхождения и
не могут разойтись по выборкам. Четыре пары дают восемь обучающих документов, одна
пара — два валидационных, последняя — два тестовых. Это маленький проверяемый
пример, а не универсальная рекомендация по долям разбиения.

## Полнота и непересечение в одной записи

Инвариант разбиения записывается так:

$$
\mathcal{D}=\mathcal{D}_{tr}\mathbin{\dot\cup}\mathcal{D}_{va}\mathbin{\dot\cup}\mathcal{D}_{te},\quad \mathcal{D}_{a}\cap\mathcal{D}_{b}=\varnothing\;(a\ne b)
$$

Знак дизъюнктного объединения одновременно утверждает две вещи: три выборки
покрывают весь корпус, и их элементы не повторяются. Условие с пересечением ещё
раз явно фиксирует второе требование для любой пары разных ролей. Только обучающая
выборка вправе задавать слияния BPE и параметры модели. Валидационная позже может
влиять на выбор, а тестовая остаётся закрытой до итогового отчёта.

Одной формулы недостаточно для программного контракта. Она не говорит о непустых
выборках, неизменности байтов, неизвестных ID, исходном порядке, группах
происхождения и репрезентативности данных. Эти свойства Rust проверяет отдельно.

{/* chapter-section:symbol-glossary */}

## Обозначения

| Обозначение | Значение |
| :--- | :--- |
| $\mathcal{D}$ | корпус как множество целых исходных документов |
| $\mathcal{D}_{tr}$ | обучающие документы — единственный источник статистик, вычисляемых по обучающим данным |
| $\mathcal{D}_{va}$ | валидационные документы, зарезервированные для последующего выбора настроек и контрольной точки |
| $\mathcal{D}_{te}$ | тестовые документы, зарезервированные для итогового отчёта |
| $tr,va,te$ | обозначения обучающей, валидационной и тестовой выборок |
| $\dot\cup$ | дизъюнктное объединение: каждый документ включён и ни один не повторяется |
| $a,b$ | любые два обозначения выборок из $tr$, $va$ и $te$ |
| $\cap$ | пересечение множеств — документы, общие для двух выборок |
| $\varnothing$ | пустое множество, то есть отсутствие общих документов |
| $a\ne b$ | условие пересечения сравнивает только разные выборки |

В формуле намеренно нет процентов. Подходящая доля зависит от объёма данных и
цели оценки. Здесь инвариантом служит точное однократное назначение, а соотношение
`8 / 2 / 2` относится только к зафиксированному учебному примеру.

## До надёжных границ отложенных данных

Оценивать модель на тексте, использованном для обучения, было удобно, однако такая
оценка отвечает лишь на вопрос, насколько хорошо система описывает уже виденные
наблюдения. Она не даёт независимых сведений о новом тексте. Перекрёстная проверка
и фиксированные отложенные выборки разделили обучение и оценивание предсказаний;
в языковом моделировании дополнительно приходится правильно выбрать границу
исходного текста.

Небезопасная последовательность выглядит вполне обычно: создать много частично
перекрывающихся фрагментов, присвоить им новые ID, а затем случайно распределить
эти ID. Исполняемый пример берёт строку `north star glows softly` и строит два
трёхсловных окна. Позиции и имена у них различаются, но `star glows` встречается в
обоих:

Историческую линию показывают первичные источники. В статье Стоуна 1974 года
формализованы
[перекрёстный выбор и оценивание](https://doi.org/10.1111/j.2517-6161.1974.tb00994.x).
Позднее Мерити и соавторы представили корпус для языкового моделирования с длинным
контекстом [WikiText](https://arxiv.org/abs/1609.07843), а Браун и соавторы включили
в работу о большой языковой модели отдельный
[анализ загрязнения обучающих данных](https://arxiv.org/abs/2005.14165).

Назначение целых документов предотвращает именно этот путь утечки. Оно не
доказывает репрезентативность корпуса, а идентификатор не обнаруживает все копии и
почти одинаковые фрагменты. Валидационная выборка тоже перестаёт быть нетронутой,
как только влияет на решение. В протоколе этого курса лишь тестовая выборка
зарезервирована для итоговой оценки.

## Закрепите границу в Rust

ошибочный UTF-8 остаётся наблюдаемым. Явные маркеры `%% document` и `%% end`
сохраняют единицу разбиения. Загрузчик отклоняет незакрытые границы, пустые тела,
повторяющиеся ID и полностью одинаковые тела документов, сохраняя исходный
порядок, язык и группу происхождения:

достоверного происхождения.

Парсер манифеста понимает только узкую схему JSON этого репозитория. Сам разбор —

## Проверьте все двенадцать назначений одним взглядом

Три области ниже — равноправные множества, а не этапы конвейера слева направо.
Каждая карточка обозначает один неделимый исходный документ. Значки, заголовки,
счётчики, рамки, стабильные ID и итоговая сводка передают роль без опоры на цвет.
Английский и русский варианты одной сцены имеют общую метку происхождения и
находятся в одной области.

Диаграмма строится непосредственно из того же корпуса и файла `splits.json`, что
использует Rust. Статическая сборка отклонит повторное, пропущенное, неизвестное,
переставленное или разделяющее происхождение назначение. На узком экране области
располагаются друг под другом, но состав и исходный порядок не меняются.

## Передайте BPE только обучающие документы

Глава 3 получает `partitions.training_documents()`: восемь заимствованных целых

Счётчик соседних байтов будет сбрасываться на каждой границе документа, поэтому ни

Каждый вариант токенизатора или модели вычисляет правила нормализации, статистику
слияний, словарь и веса только по обучающим документам. Валидационная выборка может
сравнивать такие обученные без неё варианты и выбирать контрольную точку, но не
даёт данных для подсчётов или градиентов. Тестовая выборка не участвует ни в
обучении, ни в выборе и используется только для итогового отчёта о функции потерь.
Такая граница ролей сохраняется во всех последующих главах — от BPE до итогового
сравнения декодера.
