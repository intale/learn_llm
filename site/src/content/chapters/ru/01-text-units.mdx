## Начните с двух входов по три буквы

Rust хранит и `cat`, и `кот` в UTF-8, но одинаковая видимая длина не означает
одинаковое число байтов. Прежде чем смотреть в таблицу, предскажите для каждого
входа три числа: количество байтов, скалярных значений Unicode и идентификаторов
словаря.

| Представление | `cat` | `кот` |
| :--- | :--- | :--- |
| Байты UTF-8 | `[99, 97, 116]` | `[208, 186, 208, 190, 209, 130]` |
| Скалярные значения Unicode | `[U+0063, U+0061, U+0074]` | `[U+043A, U+043E, U+0442]` |
| Идентификаторы словаря | `[3, 2, 4]` | `[5, 6, 7]` |
| Декодированный текст | `cat` | `кот` |

Для `cat` получается `3 / 3 / 3`, а для `кот` — `6 / 3 / 3`. Каждое
кириллическое скалярное значение здесь занимает два байта UTF-8, но оба входа
по-прежнему содержат три скалярные позиции и дают три идентификатора.

Идентификаторы берутся из одной фиксированной обучающей строки `cat кот`.
Идентификатор `0` зарезервирован для `<UNK>`; известные скалярные значения
сортируются по числовому значению Unicode и получают идентификаторы начиная с `1`:

| Идентификатор | Единица | Значение Unicode |
| ---: | :---: | :--- |
| 0 | `<UNK>` | неизвестная единица |
| 1 | пробел | `U+0020` |
| 2 | `a` | `U+0061` |
| 3 | `c` | `U+0063` |
| 4 | `t` | `U+0074` |
| 5 | `к` | `U+043A` |
| 6 | `о` | `U+043E` |
| 7 | `т` | `U+0442` |

Это детерминированное правило: оно не зависит от порядка обхода хеш-таблицы.
Для известных единиц прямое и обратное преобразование восстанавливает исходный
текст. У незнакомого входа `?` байт `63` и скалярное значение `U+003F` видны до
поиска в словаре, но кодирование даёт `[0]`, а декодирование возвращает `<UNK>`,
не исходный вопросительный знак.

## Одно обращение к словарю для каждой позиции

Для каждой скалярной позиции кодирование выполняет одно и то же обращение к
словарю:

$$
z_i = V(u_i), \quad u_i \notin S \Rightarrow V(u_i)=0
$$

Иными словами, возьмите скалярное значение в позиции `i`, найдите его в
фиксированном словаре и поместите полученный целочисленный идентификатор в ту же
позицию. Скалярное значение может занимать несколько байтов UTF-8, но в
последовательности токенов этой главы ему соответствует ровно одна позиция.
Обратный поиск восстанавливает каждое известное скалярное значение;
идентификатор `0` намеренно восстанавливает только маркер `<UNK>`.

{/* chapter-section:symbol-glossary */}

## Обозначения

| Обозначение | Значение |
| :--- | :--- |
| $u_i$ | скалярное значение Unicode во входной позиции $i$ |
| $S$ | фиксированное множество известных скалярных значений Unicode |
| $V$ | детерминированное отображение скалярного значения в идентификатор словаря |
| $z_i$ | идентификатор токена в позиции $i$ последовательности |
| $i$ | позиция, отсчитываемая от нуля, в последовательностях скалярных значений и токенов |
| $0$ | зарезервированный идентификатор словаря для `<UNK>` |

## До субсловных токенизаторов

Один из ранних и наглядных подходов — словарь целых слов, разделённых пробельными
символами: `red fox` превращается в `["red", "fox"]`. Такой подход легко
объяснить, но для каждого незнакомого написания, новой словоформы или слова с
присоединённым знаком препинания нужна новая запись; иначе форма становится
неизвестной.

Посимвольные модели перешли к гораздо меньшим единицам. В этой главе выражение
«посимвольный уровень» означает именно скалярные значения Unicode, которые
возвращает `str::chars` в Rust, а **не** воспринимаемые пользователем графемные
кластеры. При скалярном разбиении неизвестность на уровне целого слова исчезает,
но последовательности становятся длиннее, а одна видимая графема по-прежнему
может содержать несколько скалярных значений.

зафиксируем разбиение данных; в третьей выучим правила слияния BPE, а в четвёртой
применим готовый токенизатор.

Диаграмма группирует байты, принадлежащие одному скалярному значению, а затем
сохраняет соответствие этого значения итоговому идентификатору. Читайте

Обратите внимание на структурный инвариант: в обоих примерах три группы

последнем этапе стоял бы зарезервированный идентификатор `0`.

## Граница данных для декодера

Эта глава добавляет детерминированную границу `String -> Vec<usize>`. В общей
модели каждый `usize` будет выбирать одну строку таблицы эмбеддингов, прежде чем
последовательность поступит в декодер. Затем декодер предскажет следующий
идентификатор, а обратный словарь превратит сгенерированные идентификаторы обратно
в текст.

Во второй главе мы сохраним границы документов и зафиксируем разбиение данных.
В третьей главе выучим правила слияния BPE, а в четвёртой применим их, заменив
скалярные единицы субсловными и сохранив интерфейс целочисленной
последовательности. Поэтому все последующие части — эмбеддинги, внимание и
предсказание следующего токена — смогут работать с идентификаторами, не зная,
сколько байтов UTF-8 породило каждый из них.
